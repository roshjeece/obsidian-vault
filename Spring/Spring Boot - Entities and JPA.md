# Spring Boot - Entities and JPA

BLUF: An entity is a Java class that maps to a database table. JPA (Java Persistence API) is the specification that defines how that mapping works, and Hibernate is the library that implements it. You write Java — Hibernate writes SQL.

---

## What Is JPA?

JPA is not a library you download — it's a **specification** (a set of rules). Hibernate is the most common **implementation** of that specification. When you add `spring-boot-starter-data-jpa` to `build.gradle`, you're getting Hibernate under the hood.

```
Your Java Code
      ↓
JPA Annotations (the rules)
      ↓
Hibernate (the engine that follows the rules)
      ↓
JDBC (raw database connection)
      ↓
PostgreSQL
```

## Category Entity

```java
// Category.java
@Entity
@Table(name = "category")
public class Category {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String label;

    public Category() {}                        // required by JPA — do not remove
    public Category(String label) { this.label = label; }

    // getters and setters...
}
```

**Line-by-line:**

`@Entity` — registers this class with JPA. Hibernate now knows this class represents a row in a table.

`@Table(name = "category")` — maps to the `category` table in your database. Matches V1__init.sql exactly:
```sql
CREATE TABLE category (
    id    BIGINT GENERATED BY DEFAULT AS IDENTITY NOT NULL,
    label TEXT,
    CONSTRAINT pk_category PRIMARY KEY (id)
);
```

`@Id` — this field is the primary key.

`@GeneratedValue(strategy = GenerationType.IDENTITY)` — the database generates the id. Maps to `GENERATED BY DEFAULT AS IDENTITY` in the SQL. You never set `id` manually — you save the object and the database assigns it.

`private String label` — no annotation needed. JPA maps this to a column named `label` automatically. Field name = column name by default.

**The no-arg constructor** (`public Category() {}`) is required by JPA. Hibernate creates objects by calling the no-arg constructor, then setting fields via reflection. If you remove it, JPA crashes.

## Task Entity

```java
// Task.java
@Entity
public class Task {

    @Id
    @GeneratedValue           // uses GenerationType.AUTO — picks the task_seq sequence
    private Long id;

    private String title;
    private String description;
    private Boolean isComplete;

    @ManyToOne
    @JoinColumn(name = "category_id")
    private Category category;

    public Task() {}
    public Task(String title, String description, Boolean isComplete, Category category) { ... }

    // getters and setters...
}
```

**Key differences from Category:**

`@GeneratedValue` (no strategy) defaults to `AUTO`. Spring Boot with PostgreSQL will use the `task_seq` sequence defined in V1__init.sql:
```sql
CREATE SEQUENCE IF NOT EXISTS task_seq START WITH 1 INCREMENT BY 50;
```
The `INCREMENT BY 50` is Hibernate's default sequence allocation size — it pre-allocates 50 ids at a time for performance.

`@Table` is omitted — JPA defaults to the class name as the table name, so `Task` maps to `task`. This matches the table in V1__init.sql.

## The Relationship: @ManyToOne

```java
@ManyToOne
@JoinColumn(name = "category_id")
private Category category;
```

This is the most important thing in Task.java. It defines a relationship between two tables.

**What it means in English:** Many Tasks can belong to one Category.

**What it generates in SQL:** A `category_id` foreign key column in the `task` table that references the `category` table's `id`. This matches:
```sql
ALTER TABLE task
    ADD CONSTRAINT FK_TASK_ON_CATEGORY FOREIGN KEY (category_id) REFERENCES category (id);
```

**How it works at runtime:**
When you load a `Task` from the database, Hibernate automatically runs a second query to fetch the associated `Category` and populates `task.category` with the full object. You don't write that join — Hibernate does it.

**What this looks like as JSON when returned from the API:**
```json
{
  "id": 1,
  "title": "Learn TDD",
  "description": "research TDD",
  "isComplete": false,
  "category": {
    "id": 1,
    "label": "started"
  }
}
```

The whole `Category` object is nested inside `Task` — not just the `category_id` number.

## Getters, Setters, and the Fluent Pattern

Most of Task's setters return `this`:

```java
public Task setDescription(String description) {
    this.description = description;
    return this;    // returns the Task itself
}

public Task setIsComplete(Boolean isComplete) {
    this.isComplete = isComplete;
    return this;
}
```

This is the **fluent builder pattern** — it allows chaining:
```java
task.setDescription("new desc").setIsComplete(true);
```

Compare to Category's setters, which return `void` — standard style. Both work. The fluent style is a deliberate choice for readability when setting multiple fields.

---

## Column Naming: isComplete vs is_complete

Your Java field is `isComplete` (camelCase). Your SQL column is `is_complete` (snake_case).

Spring Boot auto-configures Hibernate's naming strategy to convert camelCase → snake_case automatically. No annotation needed. `isComplete` → `is_complete`. This is why it just works.

---

## What JPA Does NOT Do

JPA does not:
- Create your tables (Flyway does that via V1__init.sql)
- Validate that your entity matches the schema at design time (only at startup)
- Protect you from schema drift if someone edits the SQL file and doesn't update the entity

Your entity and your Flyway migration must stay in sync manually. If you add a field to `Task.java`, you need a new migration file (e.g., `V2__add_due_date_to_task.sql`).
