# Flyway

BLUF: Flyway is a database migration tool that tracks and applies versioned SQL scripts automatically at application startup. It solves the problem of keeping your database schema in sync across every environment — local, test, staging, production — without manual SQL execution. Every schema change is a file in version control.

---

## What Problem It Solves

Without Flyway, schema changes require someone to manually run SQL against each environment. This creates drift — your local database ends up different from production, or a teammate's machine doesn't have a column you added last week. Flyway eliminates this by making schema changes code: versioned, tracked, and automatically applied.

**The contract:** you write SQL migration files. Flyway runs them in order on startup, tracks which ones have already run, and skips them on subsequent startups. The database always reflects exactly the migrations that exist in your codebase.

---

## How It Works

```
App starts
    │
    ▼
Flyway checks flyway_schema_history table
    │
    ├── Table doesn't exist → create it, run all migrations from V1
    │
    └── Table exists → compare applied migrations to files in db/migration/
            │
            ├── New files found → apply them in version order
            │
            └── No new files → do nothing, app continues starting
```

Flyway creates and maintains a `flyway_schema_history` table in your database. Every applied migration gets a row with its version, description, checksum, and execution timestamp. You can query this table directly to see exactly what has run.

```sql
SELECT * FROM flyway_schema_history;
```

## File Location and Naming Convention

```
src/main/resources/db/migration/
├── V1__init.sql
├── V2__add_due_date_to_task.sql
├── V3__add_index_on_category_label.sql
```

**Naming rules — all four parts are required:**

```
V{version}__{description}.sql
│  │         │              │
│  │         │              └── must be .sql
│  │         └── double underscore (not single)
│  └── integer, must be higher than the last migration
└── capital V
```

|Part|Rule|Example|
|---|---|---|
|`V`|Capital V, required|`V`|
|version|Integer, sequential, no gaps required but no going backward|`1`, `2`, `1_1`|
|`__`|Double underscore separator|`__`|
|description|Snake_case, describes the change|`init`, `add_due_date_to_task`|

**Your actual migration:**

```sql
-- V1__init.sql
CREATE SEQUENCE IF NOT EXISTS task_seq START WITH 1 INCREMENT BY 50;

CREATE TABLE category
(
    id    BIGINT GENERATED BY DEFAULT AS IDENTITY NOT NULL,
    label TEXT,
    CONSTRAINT pk_category PRIMARY KEY (id)
);

CREATE TABLE task
(
    id          BIGINT NOT NULL,
    title       TEXT,
    description TEXT,
    is_complete BOOLEAN,
    category_id BIGINT,
    CONSTRAINT pk_task PRIMARY KEY (id)
);

ALTER TABLE task
    ADD CONSTRAINT FK_TASK_ON_CATEGORY FOREIGN KEY (category_id) REFERENCES category (id);
```

Note: `BIGINT GENERATED BY DEFAULT AS IDENTITY` is the SQL standard syntax for an auto-incrementing column. This is what your project uses — not `SERIAL`, which is older PostgreSQL shorthand for the same concept. Don't mix them.

## Gradle Dependencies

```groovy
implementation 'org.springframework.boot:spring-boot-starter-flyway'
implementation 'org.flywaydb:flyway-core'
implementation 'org.flywaydb:flyway-database-postgresql'
```

`flyway-database-postgresql` is required in Flyway 10+ (which Spring Boot 4.x pulls in). Earlier versions only needed `flyway-core`. If you're on Spring Boot 3.x, `flyway-core` alone may be sufficient.

## application.yaml Configuration

```yaml
spring:
  flyway:
    enabled: true
    locations: classpath:db/migration
```

`classpath:db/migration` tells Flyway to look in `src/main/resources/db/migration/`. This is the default location — you only need to specify it explicitly if you're changing it.

## Flyway and JPA: Who Owns the Schema

Your project uses both Flyway and JPA (Hibernate). They can conflict if not configured correctly.

**Flyway owns the schema.** It creates and modifies tables via SQL migration files. This is the correct approach — your schema is explicit, version-controlled, and reproducible.

**JPA's `ddl-auto` must be set appropriately.** Your current `application.yaml` has:

```yaml
jpa:
  hibernate:
    ddl-auto: update
```

`update` tells Hibernate to try to modify the schema to match your entities. This conflicts with Flyway's ownership of the schema. The correct setting when using Flyway is:

```yaml
jpa:
  hibernate:
    ddl-auto: validate   # Hibernate checks entities match schema, but changes nothing
```

With `validate`, Hibernate will fail at startup if your entity doesn't match the table — which is useful. With `update`, Hibernate silently modifies tables, which undermines the point of having migration files.

**The entity and the migration must stay in sync manually.** If you add a field to `Task.java`, you must also write a new migration:

```sql
-- V2__add_due_date_to_task.sql
ALTER TABLE task ADD COLUMN due_date DATE;
```

There is no automation here. Forgetting to write the migration means the app starts (with `update`) but the column doesn't exist in production (where `validate` is used), causing a failure.

## Flyway in Tests

`@DataJpaTest` uses H2 (in-memory database) by default. **H2 does not run your Flyway migrations.** Instead, it generates the schema directly from your JPA entity annotations.

This means your `@DataJpaTest` tests are not testing your SQL migrations — they're testing your entity mappings. This is intentional: repository tests are fast and isolated, not full-stack.

```groovy
// build.gradle — H2 is testRuntimeOnly, not in production
testRuntimeOnly 'com.h2database:h2'
```

If you want to test that your Flyway migrations actually work correctly, use `@SpringBootTest` with your real PostgreSQL running via Docker. The integration tests in `TaskControllerIntegrationTest` do exactly this.

**If you ever need `@DataJpaTest` to run against real PostgreSQL with Flyway:**

```java
@DataJpaTest
@AutoConfigureTestDatabase(replace = AutoConfigureTestDatabase.Replace.NONE)
class MyRepositoryTest { ... }
```

This opts out of H2 substitution and uses your configured datasource instead. Your project doesn't currently do this — it's an option for when you need to test migration-specific behavior.

## Common Failure Modes

**Checksum mismatch:**

```
FlywayException: Validate failed: Migration checksum mismatch for migration version 1
```

Cause: you edited `V1__init.sql` after it had already been applied. Flyway stores a checksum of every applied file. If the file changes, the checksum no longer matches and Flyway refuses to start.

Fix: never edit an applied migration. Create a new one instead (`V2__fix_something.sql`).

**Migration out of order:** Flyway applies migrations in version order. If `V3` exists but `V2` doesn't, Flyway will error. Keep versions sequential. If you need to insert a migration between two existing ones, use sub-versions: `V1_1__something.sql`.

**Pending migration on a clean database:** If you run the app against a fresh database with no `flyway_schema_history` table, Flyway runs all migrations from `V1` in order. This is the correct behavior and how you set up a new environment.

**Test database accumulating state:** If you run `@SpringBootTest` integration tests without `@Transactional`, data written during tests persists in the database. Your `TaskControllerIntegrationTest` uses `@Transactional` to roll back after each test method, keeping the database clean.
