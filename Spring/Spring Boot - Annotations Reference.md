# Spring Boot - Annotations Reference

BLUF: Annotations are instructions to the Spring framework written directly in your code. They replace hundreds of lines of XML configuration that older Spring apps required. Every `@` symbol in your project has a specific job — this note documents all of them from your codebase.

---

## Application-Level Annotations

### `@SpringBootApplication`
```java
@SpringBootApplication
public class TodoApplication { ... }
```
Applied to your main class. Shorthand for three annotations combined:
- `@SpringBootConfiguration` — marks this as a source of bean definitions
- `@EnableAutoConfiguration` — activates Spring Boot's auto-config based on classpath
- `@ComponentScan` — scans this package and all sub-packages for Spring components

**You only ever put this on one class per application.**

---

## Stereotype Annotations (Component Registration)

These tell Spring "create an instance of this class and manage it." Spring calls these managed instances **beans**.

### `@RestController`
```java
@RestController
@RequestMapping("/api/v1/task")
public class TaskController { ... }
```
Marks this class as an HTTP request handler. Combines `@Controller` + `@ResponseBody`, meaning every method's return value is automatically serialized to JSON (via Jackson) and written to the HTTP response body.

### `@Service`
```java
@Service
public class TaskService { ... }
```
Marks this class as a business logic component. Functionally identical to `@Component` — the distinction is semantic, making your intent clear to other developers and enabling targeted tooling/AOP.

### `@Repository`
```java
@Repository
public interface TaskRepository extends JpaRepository<Task, Long> { ... }
```
Marks this as a data access component. Also enables Spring's exception translation — database-specific exceptions (like Hibernate exceptions) are automatically converted to Spring's consistent `DataAccessException` hierarchy.

## HTTP Mapping Annotations

These live on controller methods and tell Spring which HTTP verb + URL path routes to that method.

| Annotation | HTTP Verb | Typical Use |
|---|---|---|
| `@GetMapping` | GET | Read/retrieve data |
| `@PostMapping` | POST | Create new data |
| `@PutMapping` | PUT | Replace existing data |
| `@PatchMapping` | PATCH | Partially update data |
| `@DeleteMapping` | DELETE | Remove data |

```java
@RequestMapping("/api/v1/task")   // base path for the whole controller
public class TaskController {

    @PostMapping                   // → POST  /api/v1/task
    public Task saveNewTask(...) { ... }

    @GetMapping()                  // → GET   /api/v1/task
    public List<Task> findAllTasks() { ... }

    @GetMapping("/{id}")           // → GET   /api/v1/task/42
    public Task findTaskById(...) { ... }
}
```

## Method Parameter Annotations

### `@RequestBody`
```java
public Task saveNewTask(@RequestBody Task task) { ... }
```
Tells Spring to deserialize the HTTP request body (JSON) into the specified Java object. Jackson handles the conversion automatically based on field names matching JSON keys.

### `@PathVariable`
```java
@GetMapping("/{id}")
public Task findTaskById(@PathVariable Long id) { ... }
```
Extracts a value from the URL path. The `{id}` in the `@GetMapping` path maps to the `id` parameter. A request to `/api/v1/task/42` sets `id = 42L`.

### `@ResponseStatus`
```java
@PostMapping
@ResponseStatus(HttpStatus.CREATED)
public Task saveNewTask(@RequestBody Task task) { ... }
```
Sets the HTTP status code of the response. Without this, successful responses default to `200 OK`. `HttpStatus.CREATED` sends `201 CREATED`, which is the correct status for resource creation.

## JPA / Persistence Annotations

### `@Entity`
```java
@Entity
public class Task { ... }
```
Marks this class as a JPA entity — tells Hibernate this class maps to a database table. Required on every class that represents a database row.

### `@Table`
```java
@Entity
@Table(name = "category")
public class Category { ... }
```
Specifies the exact table name. If omitted, JPA uses the class name (so `Task` maps to table `task` by default). Use `@Table` when the class name and table name differ.

### `@Id`
```java
@Id
@GeneratedValue(strategy = GenerationType.IDENTITY)
private Long id;
```
Marks the field as the primary key. Every entity must have exactly one `@Id` field.

### `@GeneratedValue`
Tells JPA how to generate the primary key. Two strategies used in your project:

| Strategy | Behavior | Used on |
|---|---|---|
| `GenerationType.IDENTITY` | Database auto-increments (uses `GENERATED BY DEFAULT AS IDENTITY`) | `Category` |
| `GenerationType.AUTO` (default) | JPA picks a strategy — uses the `task_seq` sequence from V1__init.sql | `Task` (implied by `@GeneratedValue` alone) |

### `@ManyToOne`
```java
@ManyToOne
@JoinColumn(name = "category_id")
private Category category;
```
Defines a many-to-one relationship: many Tasks can belong to one Category. The `@JoinColumn` specifies the foreign key column in the `task` table that holds the Category's id. This matches the `FK_TASK_ON_CATEGORY` constraint in V1__init.sql.

## Test Annotations

### `@SpringBootTest`
```java
@SpringBootTest
class TodoApplicationTests { ... }
```
Loads the **full** application context — every bean, every config, every database connection. Used for integration tests. Slower but most realistic.

### `@WebMvcTest`
```java
@WebMvcTest(TaskController.class)
class TaskControllerTest { ... }
```
Loads **only** the web layer (controllers, filters, Jackson). Does not load services, repositories, or the database. Services must be mocked with `@MockitoBean`. Fast — good for testing HTTP routing, request/response serialization, and status codes.

### `@DataJpaTest`
```java
@DataJpaTest
class TaskRepositoryTest { ... }
```
Loads **only** the JPA layer — repositories and the database. Uses H2 (in-memory) instead of PostgreSQL by default, so tests don't hit your real database. Good for testing custom queries and repository methods.

### `@AutoConfigureMockMvc`
```java
@SpringBootTest
@AutoConfigureMockMvc
public class TaskControllerIntegrationTest { ... }
```
Used with `@SpringBootTest` to inject a `MockMvc` instance that can fire HTTP requests through the full stack without starting a real HTTP server.

### `@Transactional`
```java
@SpringBootTest
@Transactional
public class TaskControllerIntegrationTest { ... }
```
On a test class, this rolls back all database changes after each test method. Keeps your test database clean without manually deleting data in `@AfterEach`.

### `@ExtendWith(MockitoExtension.class)`
```java
@ExtendWith(MockitoExtension.class)
class TaskServiceTest { ... }
```
Activates Mockito annotations (`@Mock`, `@InjectMocks`) for a plain unit test — no Spring context loaded at all. Fastest test type.

### `@Mock` and `@InjectMocks`
```java
@Mock
private TaskRepository taskRepository;

@InjectMocks
TaskService taskService;
```
`@Mock` creates a fake stand-in for a dependency. `@InjectMocks` creates the real object being tested and injects all `@Mock` fields into its constructor. Together they let you test the service in isolation without a database.

### `@MockitoBean`
```java
@MockitoBean
TaskService taskService;
```
Used inside `@WebMvcTest`. Registers a Mockito mock as a Spring bean in the test context. This is Spring Boot 4's replacement for the older `@MockBean`.

### `@Captor`
```java
@Captor
ArgumentCaptor<Task> captor = ArgumentCaptor.forClass(Task.class);
```
Captures the actual argument passed to a mocked method so you can make assertions on it. Lets you verify not just that a method was called but what it was called with.

### `@BeforeEach`
```java
@BeforeEach
void setUp() {
    started = new Category("started");
    learnMock = new Task(...);
}
```
Runs before every test method in the class. Used to set up test data and avoid repeating setup code in every test.

### `@Autowired`
```java
@Autowired
private MockMvc mockMvc;
```
Tells Spring to inject the matching bean from the application context. Used in tests (and occasionally in production code, though constructor injection is preferred in production classes).
